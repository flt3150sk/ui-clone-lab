# Orchestration Creator

Sub-agent型オーケストレーションスキルを設計・生成するメタスキル。
ユーザーのワークフロー要件をヒアリングし、SKILL.md + agents/ + references/ + scripts/ の一式を生成する。

## 設計思想

このスキル自体が Sub-agent型オーケストレーションの実例になっている。

- **SKILL.md**（本ファイル）: オーケストレーター。フロー制御のみ行い、専門処理はサブエージェントに委譲する
- **agents/**: 各フェーズの専門家プロンプト（設計・生成・検証）
- **references/**: パターンカタログとスキーマ定義
- **scripts/**: ディレクトリ構造の scaffolding など確定的処理

コンテキストウィンドウは公共財。各フェーズで必要な情報だけを載せる Progressive Disclosure を徹底する。

---

## ワークフロー

### Phase 0: 意図把握

ユーザーの要件を理解する。以下を明確にする:

1. **何をオーケストレーションしたいか** — どんなタスクを並列化・分割したいのか
2. **サブエージェントの候補** — どんな専門家が必要か（例: 分析、生成、検証）
3. **入出力** — 全体の入力と最終出力は何か
4. **フロー** — どのサブエージェントが並列実行可能で、どこに依存関係があるか

会話の文脈から既に答えが出ている場合は、確認だけして先に進む。

### Phase 1: アーキテクチャ設計

`agents/architect.md` を読み、サブエージェントに設計を委譲する。

**入力**: Phase 0 で得た要件
**出力**: 以下を含む設計ドキュメント
- サブエージェント一覧（名前、役割、入出力）
- 依存関係グラフ（何が並列実行可能か）
- references/ に入れるべきドメイン知識の一覧
- scripts/ に切り出すべき確定的処理の一覧

設計ドキュメントをユーザーに提示し、フィードバックを得てから次へ進む。

### Phase 2: スキル生成

設計が承認されたら、以下を並列で生成する:

1. **SKILL.md 生成** — `agents/skill-writer.md` を読んでサブエージェントに委譲。オーケストレーター本体を生成する
2. **エージェントプロンプト生成** — `agents/agent-writer.md` を読んでサブエージェントに委譲。agents/ 配下の各サブエージェントプロンプトを生成する
3. **スキーマ・リファレンス生成** — references/ 配下のスキーマ定義やパターンドキュメントを生成する
4. **スクリプト生成** — 確定的処理が必要なら scripts/ 配下のスクリプトを生成する

scaffolding スクリプトでディレクトリ構造を先に作成してから、各ファイルを並列生成する:

```bash
python scripts/scaffold.py <skill-name> <output-dir> --agents agent1 agent2 agent3
```

### Phase 3: 検証

`agents/validator.md` を読んでサブエージェントに検証を委譲する。

チェック項目:
- SKILL.md がオーケストレーターに徹しているか（専門処理を含んでいないか）
- 各サブエージェントプロンプトが自己完結しているか
- スキーマ契約が agents/ と scripts/ の間で整合しているか
- Progressive Disclosure が守られているか（SKILL.md < 500行）
- Why-driven なプロンプトになっているか（不必要な ALWAYS/NEVER がないか）

検証結果をユーザーに報告し、問題があれば Phase 2 に戻って修正する。

### Phase 4: 仕上げ

1. 生成されたスキルの全体構造をツリー表示してユーザーに提示する
2. description の最適化を提案する（BP5 に基づく「押し強め」な記述）
3. テストプロンプトを 2-3 個提案し、動作確認を促す

---

## 7つの設計原則（生成するスキルに適用）

生成するスキルには以下の原則を自動的に組み込む。これらは skill-creator の分析から抽出されたベストプラクティスであり、理由を理解した上で適用することが重要。

1. **オーケストレーターパターン**: SKILL.md は制御フローのみ。専門処理は agents/ に分離する。コンテキストウィンドウの無駄遣いを防ぐため
2. **確定的処理のスクリプト化**: LLM の解釈に依存すべきでない処理（集約、ファイル操作、API呼び出し）は scripts/ に切り出す。再現性と信頼性のため
3. **スキーマ契約**: references/ で JSON スキーマを定義し、サブエージェント間のデータ受け渡しを安定化させる。スクリプトとの連携が壊れないため
4. **Why-driven Prompt**: ルール指示より理由説明を優先する。未知のケースにも対応できるようにするため
5. **Description 最適化**: Claude はスキルをアンダートリガーしがちなので、description を積極的に書く
6. **Human-in-the-Loop**: 重要な判断ポイントでユーザーの確認を挟む。自動化しすぎない
7. **環境 Portability**: コアワークフローは環境非依存にし、実行方法だけを環境に応じて変える

- `references/patterns.md` — Sub-agent型オーケストレーションのパターンカタログ。設計フェーズで参照する
- `references/schemas.md` — 生成するスキルの構造スキーマ。検証フェーズで参照する

## エージェント

- `agents/architect.md` — Phase 1: アーキテクチャ設計の専門家
- `agents/skill-writer.md` — Phase 2: SKILL.md 生成の専門家
- `agents/agent-writer.md` — Phase 2: サブエージェントプロンプト生成の専門家
- `agents/validator.md` — Phase 3: 検証の専門家
