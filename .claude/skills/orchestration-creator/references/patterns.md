# Sub-agent型オーケストレーション パターンカタログ

設計フェーズ（Phase 1）で参照するパターン集。
アーキテクトエージェントがユーザーの要件に最適なパターンを選択するために使用する。

---

## パターン1: Fan-out / Fan-in

```
SKILL.md (オーケストレーター)
    ↓ Fan-out（並列分散）
├── agent-a（独立処理A）
├── agent-b（独立処理B）
└── agent-c（独立処理C）
    ↓ Fan-in（結果集約）
aggregator（集約処理）
```

**適用場面**: 同じ種類の処理を異なるデータに対して並列実行する場合
- 例: 複数ファイルの同時分析、複数テストケースの並列実行
- 例: 同一入力に対する複数視点からの評価

**特徴**:
- サブエージェント間に依存関係なし
- 集約処理はスクリプトに切り出しやすい（確定的処理）
- スケーラビリティが高い

---

## パターン2: Pipeline（段階的処理）

```
SKILL.md (オーケストレーター)
    ↓
Phase 1: [agent-extractor] → 中間データ
    ↓
Phase 2: [agent-transformer] → 変換データ
    ↓
Phase 3: [agent-validator] → 最終出力
```

**適用場面**: 前のフェーズの出力が次のフェーズの入力になる場合
- 例: データ抽出→変換→検証
- 例: 調査→設計→実装→レビュー

**特徴**:
- 各フェーズが完了してから次に進む
- スキーマ契約が特に重要（フェーズ間のデータ形式を厳密に定義）
- Human-in-the-Loop をフェーズ間に挟みやすい

---

## パターン3: Parallel Pipeline（並列パイプライン）

```
SKILL.md (オーケストレーター)
    ↓
Phase 1 (並列):
├── [agent-a] → データA
└── [agent-b] → データB
    ↓
Phase 2 (合流):
└── [agent-c] ← データA + データB → 最終出力
```

**適用場面**: 独立した前処理の後に、結果を統合する処理がある場合
- 例: skill-creator の with_skill / baseline 並列実行 → 比較評価
- 例: フロントエンド/バックエンドの同時生成 → 統合テスト

**特徴**:
- Fan-out / Fan-in と Pipeline の組み合わせ
- 最も一般的なオーケストレーションパターン

---

## パターン4: Evaluator-Optimizer ループ

```
SKILL.md (オーケストレーター)
    ↓
generate → evaluate → improve → (繰り返し)
    ↓
└── 終了条件を満たしたら完了
```

**適用場面**: 反復的に品質を改善する場合
- 例: コード生成→テスト→修正ループ
- 例: skill-creator の改善サイクル

**特徴**:
- ループの終了条件を明確にする（回数制限 or 品質閾値）
- 各イテレーションの結果を保存する（比較可能にする）
- Human-in-the-Loop で品質判断を外部化できる

---

## パターン5: Specialist Dispatch（専門家振り分け）

```
SKILL.md (オーケストレーター)
    ↓ 入力を分析してルーティング
├── [specialist-a] ← 条件A に該当する場合
├── [specialist-b] ← 条件B に該当する場合
└── [specialist-c] ← 条件C に該当する場合
```

**適用場面**: 入力の種類によって異なる専門家に委譲する場合
- 例: ファイル種別ごとの処理（PDF/DOCX/XLSX）
- 例: クラウドプロバイダごとのデプロイ（AWS/GCP/Azure）

**特徴**:
- SKILL.md がルーティング判断を行う
- 各専門家は独立した知識を持つ（references/ で分離）
- 新しい専門家の追加が容易

---

## パターンの組み合わせ

実際のスキルでは複数パターンを組み合わせることが多い。

**skill-creator の例**:
- Parallel Pipeline（with_skill + baseline 並列→比較）
- Evaluator-Optimizer ループ（改善サイクル）
- Specialist Dispatch（grader/comparator/analyzer の使い分け）

組み合わせる際の注意点:
- 依存関係が複雑になりすぎないよう、フェーズを明確に分ける
- 各パターンの境界でスキーマ契約を定義する
- SKILL.md が全体のフローを見渡せる粒度に保つ
